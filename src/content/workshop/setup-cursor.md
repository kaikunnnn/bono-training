## カーサーって何？なぜカーサーから始めるのか

### カーサーはAIが乗ったエディター

カーサーはエディター、つまりコードを見やすく書くためのツールです。エンジニアにとってのFigmaのようなもの、と考えると分かりやすいかもしれません。そこにAIが乗っていて、バイブコーディングができる——そういう時代に出てきたエディターです。

なぜエディターから入るのがいいかというと、**技術的なことをビジュアルで見せてくれる**からです。ヒントがないやり方（クロードコードをそのまま使う方法）に比べて、開発やコードに慣れていない人でも進めやすくなります。

### なぜAIでデザイン（バイブコーディング）を覚えるのか

理由は2つあります。

1つ目は、**実物を触って検証できる**こと。私たちがデザインをするのは、突き詰めればユーザーをハッピーにするためです。Figmaでも似たことはできますが、あくまで仮のもの。実際に動くものを触って検証すると、「どんな体験が一番いいか」という気づきが一気に増えます。いいアイデアにもダメな点にも気づけて、より深く潜っていけます。

2つ目は、**非エンジニアでもコードで作れるようになった**こと。Web業界はコードで出来ています。その「コードという素材」を扱えると、自由度もクオリティも上げやすい。これが新時代のデジタルものづくりです。

### クロードコードとの違い（両方使う）

「クロードコードのほうが有名では？」と思う人もいるはずです。結論、**どちらも使います**。違いはこうです。

- **カーサー**＝エディター。コードを可視化してくれる場所に、AIが乗っている
- **クロードコード**＝エージェント（ChatGPTやGeminiのようなもの）。ターミナルさえあればどこでも呼び出せる。自律的に動いて優秀だと評価されている

クロードコードはカーサー上でも呼び出せます。ただ、まずはエディターを使えたほうがコードへの耐性がぐっと下がるので、初心者はカーサーから入るのがおすすめです。クロードコードの入門は別記事で扱います。

## ① UIをつくる

### Cursorをダウンロードして開く

まだの方は「[cursor.com](https://cursor.com)」にアクセスし、ダウンロードボタンから自分のOS（Mac / Windows / Linux）に合ったものを入れてください。インストールして開き、初期画面が出るところまで進めます。

### プロジェクト＝フォルダを用意する

ここがクラウドサービスと違って迷いやすいところです。エディターは**フォルダ／ファイルを開いて編集する**ツールで、メモ帳に近い感覚です。パソコンの中（ローカル）に保存・編集していきます。そして**プロジェクトは基本的に「フォルダ」単位**です。

新しく始めるときは、デスクトップや書類の中に新しいフォルダを作りましょう（名前は何でもOK）。カーサーの「Open Project」からそのフォルダを開けば、そこが作業場所になります。

### チャットでUIを生成する（新規登録画面）

白い画面が開くと、チャット欄があります。ChatGPTのように「◯◯を作って」「これを修正して」と会話していきます。最初なので、簡単なものを作ってみましょう。

指示のポイントは、**条件をきちんと与えること**。意図したものができやすくなります。たとえば新規アカウント登録画面なら、こう書きます。

> UIのプロトタイプを作成してほしいです。新規アカウント作成の画面です。デスクトップのデザインにしてください。html、cssで作ってください。アカウント作成に必要な入力は、メールアドレス・パスワード・登録ボタン・利用規約の同意チェックボックスです。

htmlとcssは、ブラウザで動く最低限のコードの名前です。これで作るとシンプルかつ素早くでき、誰でもブラウザで開ける状態になります。

送ると、AIが「今こういう作業をしています」と書きながら進めてくれます。途中で意図と違うと思ったら、**黒丸の停止ボタン**で中断できます。

### 出来たものを開いて触る

完成すると、2つのファイル（例：signup.html と signup.css）が作られ、使い方も説明してくれます。エディター上では直接表示できないので、書いてあるとおり**ファインダーやブラウザで signup.html を開きます**。

作られたファイルは、最初に開いたフォルダの中に追加されています。Macならファインダーからそのファイルをダブルクリックすればブラウザで開きます。メール・パスワード・利用規約など、指定したものが**実際に触れるプロトタイプ**になっているはずです。

登録後の画面は指示していないので動きませんが、こうして触れることで「実際の操作感」を確認できます。これが実物を触るパワーです。

## ② カーサーの画面と基本操作

### 主要な3ブロック（チャット／ファイル／中身）

カーサーのメインは3つのブロックです（本当はもう1つターミナルがありますが、ここでは省きます）。

- **チャット欄**：いちばん使う場所
- **ファイル構造**（右）：開いたフォルダの中身が全部出る。ファインダーで見ているのと同じもの
- **中身の表示**（真ん中）：選んだファイルの中身を開くエリア。ここで編集もできる

コードが分かる人は中身を直接編集し、Cmd+Sで保存すればプロトタイプに反映されます。また、右上のアイコンから、サイドバーやターミナル、エージェント専用ウィンドウなどの**表示を切り替え**られます。

### プランモードで作る（ログイン画面）

次にログイン画面を作りますが、ここで**モードの切り替え**を紹介します。チャット欄では、エージェント／プラン／デバッグ／アスクなどを切り替えられます。基本はエージェント（内容に応じて自動判断）でOKですが、おすすめは**プランモード**です。

プランモードは**作業をせず、まず計画を立てることに専念**します。たとえばログイン画面を頼むと、現状のHTML/CSSを確認したうえで、「スタイルはどうしますか？」「入力要素は何にしますか？」と、あいまいな部分を推測して質問してくれます。

分からない質問があれば「AとBを説明してください」と聞けば、噛み砕いて教えてくれます。技術は私たちより分かっているAIに聞いて計画を立ててもらう——この**プランニングがとても重要**です。「ビュープラン」で方針を確認し、良ければ**ビルド**を押すと、エージェントモードに切り替わって実装が進みます。

出来上がったら login.html を開きます。右側のファイル一覧で右クリック →「Reveal in Finder」からも開けます。新規登録とログインを行き来してみると、「今どっちの画面にいる？」と気づき、UIを工夫したくなる。触るからこそ気づける、というわけです。

## ③ 開発環境を整える（Next.js）

### 何のためにやるのか

開発環境とは、開発のための土台です。世界中の無料のプラグインや仕組みを組み込めるようになります。ここまではHTML/CSSという最低限の技術で画面を作っていましたが、いろんな機能を使うにはこの土台が要ります。今回は **Next.js** の開発環境を整えます。

目的は、この後やりたい「Figmaに持っていく」ために、**ローカルのURL（開発環境の表示用URL）を発行する**ことです。

### プロンプトでAIに構築してもらう

流れはシンプルです。**プロンプトを打つ → AIが自動で構築 → 起動の呪文を唱える → URLが立ち上がる**。プランモードで、こう伝えます。

> 開発環境を作成したいです。条件として、Next.jsで開発環境を整えたいです。開発環境を整える上で必要なものがあれば、AIが操作して入れて構いません。ただし、パソコンが壊れるようなものであれば入れないでください。気になることがあれば相談しながら進めてください。

すると「プロジェクトはどこに作りますか？」「パッケージマネージャーはどれにしますか？」などと聞かれます。作業中のフォルダを指定し、迷ったらパッケージマネージャーは**npm**でOK。既存のログイン／新規登録の画面もNext.jsで表示できるように、と添えておきましょう。内容を確認してビルドします。

### ターミナルとは

ここで「ターミナルで◯◯を入れてください」と言われることがあります。ターミナルは、いわば**PCへの直通電話**。黒くて怖く見えますが、コンピューターに文字で直接指示する画面です。

もともとパソコンはこのコマンドライン（CLI）で操作するもので、削除ボタンなどのUIは、その裏でコマンドを走らせています。カーサーの右上のアイコンから**カーサー内のターミナル**を開けるので、最初はそこで操作すると分かりやすいです。Node.jsなど必要なものを求められたら、ここで実行します。分からなければAIに「どうインストールすればいい？」と聞きましょう。

### 起動して確認する（npm run dev）

構築が終わったら、起動の呪文を唱えます。ターミナルで **npm run dev** と打つと、サーバーが起動し、localhost のURL（例：localhost:3001）が表示されます。これで開発環境が立ち上がっている状態です。

URLをブラウザに入れる（Cmd+クリックでも開けます）と、開発中のものが表示されます。このURLは**自分のパソコン上でしか開けません**（他の人に共有しても開きません）。Next.jsの画面にログイン／サインアップのリンクが出ていれば成功です。

## ④ カーサーのブラウザ機能で見た目を修正する

カーサーには、デザインツールのように**見た目を直接いじれるブラウザ機能**があります。

右側のコンテンツ側の右上、3点リーダーから「Open Browser」を選ぶと、ブラウザが開きます。URL欄に開発環境のURL（例：localhost:3000）を入れれば表示できます。

右上のパネルからカーソルツールをオンにして要素をクリックすると、右側にプロパティが出ます。Figmaのデザインツールのような感覚で、フォントサイズ・背景色・パディングなどを変更できます。カラーなどのトークンを定義しておけば、よりシステマチックに扱えます。

注意点として、これは**デザインツールではない**ので、要素を増やすのは実装やAIに任せ、ここでは**スタイルだけ変える**のが便利です。変更は「Apply（適用）」を押すとAIが実装を走らせて反映されます。戻したいときはUndoで戻せます。デザイナーが細かい調整やパターン作りをしたいときに向いていて、これもカーサーを使うメリットの一つです。

## ⑤ Figmaに持っていく

コードで作ったものを、そのままFigmaに持っていけます。デザイナーなら、コードとFigmaの両方でアイデアを広げたほうがいいので、いちばん簡単な方法を紹介します。使うのは **HTML to Design** というプラグインです。

流れは3ステップです。

1. **ブラウザ拡張機能を入れる**：「HTML to Design」を検索し、Chrome Web Storeからインストール。拡張機能の設定でトグルをオンにします。
2. **URLのデータをダウンロードする**：持っていきたい画面（例：新規登録）を開いた状態で拡張機能を押し、「Capture Current Page」を実行。ビューポート幅（例：1920と390でデスクトップ／スマホ両方）やテーマを選べます。.h2d というファイルがダウンロードされます。
3. **Figmaで読み込む**：Figmaで画面下の「プラグイン」から [html.design](https://html.design) を開き、ダウンロードした .h2d を**ドラッグ&ドロップ**。オプションは基本「オートレイアウト」だけオンにして「Proceed」でOK。すると画面のデータがFigmaに出てきます。

オートレイアウトが効いた状態でデータになるので、そのままFigmaで作業を続けられます。

こうして**コード（実物）とFigma（キャンバス）を行き来**しながら、リサーチ → アイデア → プロトタイプ → 評価 → 改善…とブラッシュアップしていく。これはシリコンバレーでも、AIで20〜30個のプロトタイプを一気に作って検証し、最後に一つのいい体験へまとめていく、という形で使われている進め方です。

## ⑥ Vercelで公開する（デプロイ）

最後に、作ったものをインターネットに公開します。使うのは **Vercel（バーセル）** というホスティングサービス。先ほどのNext.jsを開発している会社でもあります。

手順の大枠は「アカウント作成 → AIにお願い → 認証 → URL完成」です。

1. [**vercel.com**](https://vercel.com) でサインアップし、アカウントを作ります。
2. カーサーのチャットで、プランモードでこう聞きます。「Vercelのアカウントを作成しました。Vercelを使って今作っているものをURLで公開したいです。どうすればいいでしょうか？」
3. **GitHub連携**の話が出てきたら、今回は範囲が広がりすぎるので「GitHubは連携しない方法でいけますか？」と聞きます。連携なしで進められます。
4. 「自分でやってください」という手順が返ってきたら、「あなたが実行しながら導入するやり方を考えられませんか？」と頼みます。するとAIがターミナルで実行し、私たちは**ブラウザ操作だけ**を行う流れにしてくれます。
5. 途中で**デバイス認証**を求められたら、ブラウザで許可します。ログインが成功すれば、そのまま本番URLまで出してくれます。

Vercelのダッシュボードに戻ると、プロジェクトが増えています。開いてステータスが「Ready」なら、デプロイメントのURL（〜.vercel.app）が発行されています。クリックすると、ローカルで見ていたものと同じ画面が**本番URLとして公開**されています。これで、他の人もURLにアクセスして触れる状態になりました。インターネットに自分の作ったものが増えた、記念すべき第一歩です。

## まとめ：AIデザインの入り口に立った

今日の流れを振り返ると、UIを作り → 開発環境を整え → ログイン画面を追加し → Figmaに移し → Vercelで公開する、というところまで来ました。これで土台ができています。

ここから先は、自分の作りたい画面やフローを作ることもできますし、「こういう機能を作りたいが、どう計画を立てて進めるといい？」とAIに相談し、計画を立てて一つずつ作っていけば、サービスも形にしていけます。**AIデザインの夜明け**です。

なお、UI/UXデザインの基礎を扱うコミュニティ「BONO」も運営しています。AIに全部任せるのではなく、自分の考えを入れることでAIも的確に動く——だからこそ基礎が重要です。デザインサイクルの解説などのコンテンツを用意しているので、興味があればチェックしてみてください。（クロードコード入門、Figma MCP、GitHubの使い方などは別記事で扱う予定です。）
