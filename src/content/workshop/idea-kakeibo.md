ユーザーの困りごとから始まる依頼書ベースのお題。価値検証のための体験アイディアプロトを作る。

## 1. チャレンジ内容

家計簿アプリ『カケル』のチームの一員として、社長から「支出傾向に気づけずに困っているユーザー（2タイプ）」のための **新しい体験アイディアの検証プロト** の作成を任される。担当ペルソナを1つ選び、**ユーザーに価値が届くかを確かめる**ためのプロトをAIで作りながら、体験のアイディアを検証する。

完璧なUIを作るのではなく、**ユーザーに触ってもらって意見を引き出すための叩き台**を作るのがゴール。

## 2. 依頼内容

**CEO 西野 → あなた（Slack）**

> ちょっと相談したいことがあって。
>
> 『カケル』、リリースから半年たったけどアクティブが10人/週くらいで頭打ちでさ…。やっぱりなんの変哲もないものをAIに従って作っちゃったのがあかんのかな。
>
> ということで俺が考える、困っているユーザーが2タイプに向けてアイデアを試してみたいんだよね。（詳細は別資料で送る）。
>
> ただ、いきなり作っても外すかもしれないから、来週ユーザーヒアリングをしようと思ってる。**そのときに、会話以外に具体物を見せて意見をもらえるように準備したい**。口頭で聞くのと合わせて、イメージを触ってもらった方が違う本音を引き出せると思うので。
>
> 機能の良し悪しを聞きたいんじゃなくて、**この方向性で価値が届きそうかどうかの感触を掴みたい**。
>
> 2タイプのうちどちらか1つ選んで、ヒアリング叩き台になるプロト作ってもらえる？ 何を作るかは任せる。完璧じゃなくていい、触って意見もらえる粒度で。よろしく！

## 3. 背景情報

- **サービス名:** 『カケル』（家計簿Webアプリ）
- **運営会社:** 4人体制のスタートアップ
- **ユーザー数:** アクティブ約50人（リリース半年で頭打ち）
- **現状のUI:** 一般的な家計簿UI（カテゴリ選んで金額入れて保存、月次集計画面あり）
- **困りごと:** 反応が薄い。マネーフォワード・Zaim等の競合と差別化できていない
- **CEOの仮説:** 汎用的すぎて誰の心にも刺さってない。特化したサービスを別途立ち上げるべきかも

## 4. 救いたいユーザー

### コンビニ依存気にしてる在宅勤務会社員（30代）

> 「リモートワーク始めてから朝ごはんや昼ごはん作るのがめんどくさくて、コンビニで毎日何回も買い物してたら、月末に明細見たら4万円超えてて衝撃でした。買ってる時は全然気づかなかった…自炊したいよ...まま....」
>
> って弟が言ってたんだ。出費の自己管理の手助けができるような体験に特化できないかな？

- **困りごと:** リモートで1日2-3回コンビニ。1回300-800円が月3万円超になってるが、買ってる時は気づかない
- **やりたいこと:** 「ちりつもで月いくらコンビニなど知りたいカテゴリに払ってるか」を可視化したい

## 5. アウトプット

- [ ] **ユーザーゴールの言語化**（1枚） — ペルソナが本当に達成したいことを言葉にする
- [ ] **理想の体験フロー**（簡単な図 or 箇条書き） — 起動→気づきまでの流れ
- [ ] **ヒアリング叩き台プロトタイプ**（AIで生成した触れるWebアプリ） — Claude Code / Cursor で実装。完璧でなくていい、触って意見を引き出せる粒度
- [ ] **自己評価メモ** — 達成条件に対する自己採点 + ヒアリングで何を聞きたいか

## 6. 達成条件

「依頼を満たした」ではなく「**ユーザーに価値が届きそうかをヒアリングで確かめられる叩き台になっているか**」で判定する。

### 共通の達成条件

- [ ] **ユーザーが触って具体的な意見を返せる粒度** — 抽象的なワイヤーではなく、感想が出る形になっている
- [ ] **体験の方向性が伝わる** — 触った人が「あ、こういうの作りたいんだ」と理解できる
- [ ] **担当ペルソナのゴールに照らして、本人が「これ欲しい」と思える方向性になっている**

## 7. 身につけるデザインスキル

- **ユーザーの困りごとからゴールを言語化する力** — 「ちりつもに気づきたい」が本命であり、「家計簿をつける」は手段でしかない
- **ユーザー体験の整理フレーム** — 「やりたいこと（ゴール）→ 使うシーン → 必要な体験 → UI要素」の順で考える
- **価値検証のための叩き台プロトを作る発想** — 完璧を作るのではなく、相手の意見を引き出すための実物を素早く出す
- **プロト → 評価 → 改善のサイクル** — AIで作って終わりではなく、ペルソナ視点で触って違和感を言語化し、AIに具体指示で直してもらう

## ポイント

- 依頼書には**「何を作るか」を指定していない**。「価値が届くか確かめたい」とゴールだけ渡されている
- 参加者は「家計簿の入力UI」を作るか、「ダッシュボード」を作るか、「通知」で攻めるか、自分で発想する
- **「ヒアリング叩き台として作る」** という目的設定により、完成度より「相手の反応を引き出せるか」が成功基準になる
- 「機能を作る → 体験を作る」「完璧を作る → 叩き台で対話する」のシフトを体感できる構造
