# タスク進行の標準ループ

## 順序（必ずこの順で）

**リサーチ → 計画 → 実装 → テスト → 改善**

- リサーチは**現状実装（Explore）とデザイン（Figma）の両方**を行う。片方だけで計画を立てない

## 計画の鬼チェック（敵対的レビュー）

実装前に以下の観点で計画の穴を潰す:

- [ ] **送信/保存失敗時のUX**: 失敗時にどこに留まるか・入力は保持されるか
- [ ] **二重送信ガード** / レート制限時の文言
- [ ] **戻る・再入場での入力保持** / dirty状態の離脱ガード（確認ダイアログ + beforeunload）
- [ ] **モバイル未定義**: Figmaがデスクトップのみの場合、レスポンシブ挙動を計画に明記する
- [ ] **prefers-reduced-motion**: アニメには必ずフォールバックを付ける
- [ ] **データの真実が二重化しないか**: 定数とDBは一方向同期にする
- [ ] **参照整合**: slug→ID解決の防御。解決不能は非表示 + ログ（クラッシュも黙殺もしない）

## スライス分割

大きなタスクは**受け入れ条件付きスライス**に割り、1スライスごとに:

実装 → 静的検証（`npx tsc --noEmit` / `npm run build` / `npm run lint` / `npm test`）→ 受け入れ確認

## 実ブラウザ動作検証を省略しない

**build/tscが通っても「動く」とは言わない。**

- claude-in-chrome で実際にクリックして通す
- コンソール・`.next/dev/logs/next-development.log`・SupabaseログまでE2Eで確認する
- 実例: コメント削除のRLSバグは UPDATE+RETURNING と SELECTポリシーの衝突で、**実際に押すまで発見できなかった**（静的検証は全て通っていた）

## issueのwiki運用

- rebono/issues のMDに **決定事項 / スライス / 進捗ログ / 未決Q** を追記し続ける
- 完了時は `status: review` にし、チェックリストを**「動作確認済み（AI実施）」と「ユーザーの目視が必要」に分けて**簡潔に書く

## 環境安全

- 本番書き込み（Sanity / Supabase / Stripe）は必ず **dry-run → 内容提示 → ユーザー承認 → 実行**
- 削除は対象一覧を提示してから実行する
- 作業前に**ローカル/本番のどちらに繋がっているか**（`.env.local`）を確認する
