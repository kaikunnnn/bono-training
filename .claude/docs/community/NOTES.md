**ボタン出し分けロジック**:

**フィードバックカード**:

- テイスト: LessonCardを参考
- タイトル（必須）
- カテゴリ（必須）
- 対象アウトプット/コース名（任意、なければ非表示）
- 概要文（最大2行、必須）

### 詳細ページ `/feedback/:slug`

**構成（上から順に）**:

| #   | セクション | 内容                                                  | アクセス                           |
| --- | ---------- | ----------------------------------------------------- | ---------------------------------- |
| 1   | ヘッダー   | タイトル・カテゴリ・対象アウトプット・日付            | 全員                               |
| 2   | 動画       | Vimeo埋め込み（ある場合のみ、ブログ詳細のサムネ位置） | **課金者限定**（VideoSection使用） |
| 3   | 依頼内容   | FigmaURL（リンクのみ）・見てほしいポイント・依頼文    | 全員                               |
| 4   | 本文       | Kaiのフィードバック（リッチテキスト、ブログ風）       | **課金者限定**                     |

### Sanity Schema設計

**feedbackCategory**:

- title: string
- slug: slug

**feedback**:

- title: string（必須）
- slug: slug（必須）
- category: reference → feedbackCategory（必須）
- targetOutput: string（任意、対象コース/アウトプット名）
- excerpt: text（概要、最大2行分）
- videoUrl: url（任意、Vimeo）
- figmaUrl: url（任意）
- requestPoints: text（見てほしいポイント）
- requestContent: blockContent（依頼内容リッチテキスト）
- feedbackContent: blockContent（本文リッチテキスト）
- publishedAt: datetime

## ナビゲーションの暫定

以下のように今後していくのでその中でのお役立ち（ガイドとナレッジが合わさったもの）、と、質問、フィードバックのページを作っていく

#### 1. メイン (Main Navigation)

学習のコアとなる体験へのアクセス。

| 名称 | 役割定義 |

| :--- | :--- |

| **マイページ** | ユーザー個人のダッシュボード。進捗確認など。 |
| **トップ** | サービスのホーム画面。|
| **ロードマップ** | ゴール達成のためのメインルート（未経験転職、AIフローなど）。 |
| **コース** | ロードマップを構成する、単体の学習カリキュラム群。 |
| **トレーニング** | 手を動かす実践課題・ドリル。 |

#### 2. 探す (Explore / Resource)

目的のリソースを見つけるエリア。

| 名称 | 役割定義 |

| :--- | :--- |

| **探索** | 全体検索、またはキーワードからコンテンツを探す入り口。 |

| **お役立ち** | **(Oyaku-dachi)**<br>Tips、ガイド、コラムなどの「助けになる情報」の集合体。<br>動画・テキスト混合。 |

#### 3. コミュニティ (Community)

他者とのつながり、フィードバック。

| 名称 | 役割定義 |

| :--- | :--- |

| **みんなの作業部屋** | コワーキングスペース的な機能。 |

| **質問** | 疑問解決フォーラム。 |

| **フィードバック** | 制作物へのレビュー依頼。 |

| **Slackを開く** | 外部コミュニティ(Slack)へのリンク。 |

#### 4. サポート・設定 (Footer/System)

サービス利用の補助。

| 名称 | 役割定義 |

| :--- | :--- |

| **使い方** | サービスの利用ガイド。 |

| **BONOとは** | コンセプト、理念。 |

| **運営ブログ** | 運営からのお知らせ、更新情報。 |

| **設定** | アカウント設定、通知設定など。 |

## 📝 今日のサマリー

お疲れ様です！プロトタイプ作成まで進んだのは大きな前進です🙌

見た目を整える前に、まずは**「ユーザーが何のために使うのか？」**を整理する情報設計（IA）からやり直しましょう！ここを乗り越えれば、デザインの説得力が爆上がりします🚀

## 1. はじめに：確実な前進と次のステージへ 🔥

まず、プロトタイプの作成お疲れ様でした！！🎉
前回のフィードバックと比べて、格段に良くなっています👏
ただ「真似る」だけじゃなく、実際に動く形（プロトタイプ）まで落とし込んだのは本当にすごい進歩です！まずは「形にする力」がついたことに自信を持ってください💪
見た目のクオリティも、ベースラインとしては十分なレベルです✨

ここからは、さらに一段上のデザイナーになるための、少し厳しいけれど愛のあるフィードバックをします❤️‍🔥
それは**「なぜその形にしたのか？」という問いへの答えを持つこと**です。

## 2. 最も重要な課題：「情報設計」の不在 ⚠️

今回のデザインの最大の課題は、ズバリ…
**「画面を作る前に、情報の構造（情報設計）を整理できていないこと」** です😭

画面デザインのステップにおいて、あなたは「情報設計」のフェーズを飛ばして、いきなり「ビジュアルデザイン（ガワの作成）」に入っちゃってます💦
YouTubeなどの既存サービスを参考にする（真似る）のはめっちゃ良いこと！
でも、**「なぜYouTubeはその構造なのか？」という背景（ユースケース）を理解せずに、表面だけを真似てしまっています🙅‍♀️**

### 具体例：サイドナビゲーションの構造ミス 🚫

今回のデザインでは、「申請」という親項目の下に「下書き」「差し戻し」が並列で並んでいます。
これはYouTubeのマイページを参考にしたっぽいですが、ここには大きな決定的な違いがあります💦

| 比較項目           | ▶️ YouTube（参考元）                                      | 📱 今回の申請アプリ                                              |
| :----------------- | :-------------------------------------------------------- | :--------------------------------------------------------------- |
| **ユーザーの目的** | 「履歴を見たい」「再生リストを見たい」が頻繁にある！      | 「下書き」だけ見たいケースは稀！基本は「申請一覧」を見たいはず👀 |
| **情報の性質**     | それぞれが独立した強いコンテンツ💪 ショートカットが必要！ | 「下書き」「差し戻し」はあくまで「申請」のステータスの一つ！     |
| **あるべき構造**   | サイドバーにショートカットとして配置するのが正解⭕️        | 「申請一覧」の中に、フィルター機能やタブとして存在すべき！       |

**結論：**  
今回のアプリのユースケース（頻度や目的）を考えると、サイドバーに「下書き」を特等席として置く必要ナシ🙅‍♀️
「申請一覧」の中に、全ての情報が集約されているのが自然な情報設計です✨

## 3. その他の改善ポイント 💡

### ① 情報の優先度と視認性（ステータス） 👀

申請業務において、「差し戻し」や「承認待ち」は超重要情報！ユーザーが一番早く知りたいやつです🚨

- **現状:** ステータス表示が右端にあって見にくい…。「差し戻し」以外はアイコンのみで区別がつきにくい💦
- **改善:** 重要情報は左側（視線の始点）に近い場所に置く！あるいは色分けやラベルでパッと見て状態がわかるようにしよう！「承認待ち」なのか「完了」なのか、文字で読ませる工夫も必要です🙆‍♀️

### ② 配色（カラーセオリー）の基本 🎨

背景色に黄色系、文字やアクセントに青色を使用していますが、これは**「補色（反対色）」の関係**！
互いに反発して目がチカチカしちゃう組み合わせです😵‍💫

- **Googleの例:** Googleは青を基調とする場合、背景も青みがかったグレーにするなど、色相（色味）を統一して調和をとっています✨
- **改善:** デザインにおける配色の基礎（色相環）を学び直そう！まずは同系色でまとめるか、無彩色（グレー）をベースにすると失敗しにくいよ👍

### ③ 「思考停止」の模倣からの脱却 🤔

リストの左端にあるチェックボックスや、急に出てくる「？」アイコン…これ、「とりあえずGoogleにあるから置いた」になってない？

- **問い:** そのチェックボックスを押すと何が起きる？一括操作が必要なほど大量のデータを扱う？
- **改善:** 画面に置くすべての要素に対して「ユーザーがこれを使う理由は？」と説明できるようにしよう！理由がない要素はノイズです🙅‍♀️
