# 思考シェア記事提出ページ - UX分析と改善提案

**作成日**: 2025-02-23
**ステータス**: 分析完了・改善提案中

---

## 1. 現状分析（プロダクトオーナー視点）

### 1.1 発見された問題

| 重要度 | 問題 | 影響 |
|--------|------|------|
| 🔴 高 | カテゴリ選択でエラー発生 | ユーザーが提出できない |
| 🟡 中 | 名前フィールドが曖昧 | コミュニティ名との不一致、管理混乱 |
| 🟡 中 | BONOコンテンツが任意で下に | 記事との関連性が分かりにくい |
| 🟢 低 | フォームの順序が直感的でない | 認知負荷が高い |

### 1.2 ユーザージャーニーの課題

```
現状のフロー:
ページ到着 → ガイド読む → 名前入力 → URL入力 → 伝えたいこと → カテゴリ選択 → BONOコンテンツ → 提出

課題:
1. 「名前」が何の名前か不明確（本名？Slack名？）
2. URLを先に入力させるが、何の記事かの文脈がない
3. カテゴリ選択が最後の方にあるが、これが「判定基準」なので最初に意識させるべき
4. BONOコンテンツが「任意」で目立たない（でも運営には重要な情報）
```

### 1.3 ビジネス目標との乖離

**目標**: ユーザーに「思考の軌跡」を言語化してもらい、成長を促進する
**現実**: フォームが「提出作業」になっており、「思考の整理」になっていない

---

## 2. 改善の方向性（プロダクトマネージャー視点）

### 2.1 コアコンセプト

> 「提出フォーム」から「思考整理ツール」へ

ユーザーが記事を書いた後、このフォームで「自分が何を考えたか」を振り返り、
言語化することで、学びがさらに深まる体験を提供する。

### 2.2 改善の3つの柱

#### 柱1: 文脈優先の情報設計

**Before**: URL → 伝えたいこと → カテゴリ
**After**: BONOコンテンツ選択 → URL → 思考の振り返り

理由: 「何を学んで、何を実践したか」という文脈を先に設定することで、
ユーザーは自分の記事を客観的に見直せる。

#### 柱2: 段階的な開示（Progressive Disclosure）

1ページに全フィールドを並べるのではなく、
ステップバイステップで「思考を整理」していく体験を提供。

```
Step 1: 何を学んだ？（BONOコンテンツ選択）
Step 2: どこに書いた？（URL入力）
Step 3: 何を考えた？（カテゴリ選択 + 140字）
Step 4: 確認して提出
```

#### 柱3: アイデンティティの明確化

「名前」→「コミュニティネーム」に変更し、
アカウント設定で必須化することで、一貫性を確保。

### 3.2 カテゴリ選択のマイクロインタラクション

現状の問題: チェックボックスが「フォームの一部」になっていて味気ない

改善案: カードタップで選択状態が切り替わり、選択時にアニメーション

```tsx
// 選択前
┌──────────────────────────┐
│  ☐ 気づき・変化          │  ← グレーボーダー
│  自分の考え方が変わった   │
└──────────────────────────┘

// 選択時（タップ）
┌──────────────────────────┐
│  ✓ 気づき・変化          │  ← インディゴボーダー + 背景
│  自分の考え方が変わった   │     チェックマークにアニメーション
└──────────────────────────┘

// 複数選択可能を明示
「1つ以上選んでください（複数選択OK）」
```

### 3.3 OGPプレビューの即座表示

URL入力後、即座に記事のOGP情報を取得してプレビュー表示。
これにより「正しいURLを入力した」確認ができる。

```tsx
// URL入力中
┌─────────────────────────────────┐
│ https://note.com/user/n/abc... │
└─────────────────────────────────┘

// 入力完了後（自動取得）
┌─────────────────────────────────┐
│ [🖼️]  私のデザイン学習記録       │
│       note.com                  │
│                     ✓ 確認済み  │
└─────────────────────────────────┘
```

## 4. 技術的な修正事項

### 4.1 カテゴリ選択のバグ修正

**原因**: `onCheckedChange`の実装が不適切

```tsx
// 修正前（バグあり）
<Checkbox
  onCheckedChange={() => canSubmit && handleCategoryToggle(option.value)}
/>

// 修正後
<Checkbox
  onCheckedChange={(checked) => {
    if (canSubmit && checked !== "indeterminate") {
      handleCategoryToggle(option.value);
    }
  }}
/>
```

また、divのonClickとCheckboxのonCheckedChangeが重複しているため、
イベントのバブリングを防ぐ必要がある。

### 4.2 BONOコンテンツ選択の追加

Sanityから取得したコンテンツリストをドロップダウンまたは
オートコンプリートで選択可能にする。

## 7. 参考: Airbnb的なUXプリンシプル

1. **一度に一つのことを聞く** - マルチステップで認知負荷を下げる
2. **即座のフィードバック** - OGPプレビューで入力確認
3. **文脈を先に設定** - 「何を学んだか」から始める
4. **エラーを防ぐ** - 必須項目を明確にし、入力補助を提供
5. **達成感を与える** - ステッププログレスで進捗を可視化
