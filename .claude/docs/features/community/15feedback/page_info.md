# 思考シェア記事 提出ページ：ページ仕様書

> Airbnbのプロダクトデザイナー / プロダクトオーナーの視点で設計。
> ユーザーが「フィードバックをもらうために記事を書く」行為を、成長実感のある体験として設計する。

---

## 1. ページの目的

### ビジネスゴール

- ユーザーが思考の記事を書いて提出する → フィードバック権利を獲得する循環を生む
- 高品質な記事がnote等に蓄積される → BONOの「思考を鍛えるコミュニティ」としてのブランドが可視化される

### ユーザーゴール

- 自分の成長を記録し、プロからフィードバックをもらいたい
- でも「何を書けばいいか分からない」「どうすれば通るのか不安」

### デザイン原則（Airbnb的アプローチ）

| 原則                    | このページでの適用                                                 |
| ----------------------- | ------------------------------------------------------------------ |
| **Belong Anywhere**     | 「完璧な記事」でなくていい。思考を回した軌跡があればOKという安心感 |
| **Remove friction**     | 記事を書く→提出するまでの摩擦を最小化。迷わせない導線              |
| **Educate, don't gate** | ルールを「門番」ではなく「コーチ」として提示。通過率を上げる設計   |
| **Celebrate progress**  | 提出自体が成長の証。判定結果に関わらず肯定する体験                 |

---

## 2. ページ構成

### 全体フロー

```
[HERO] 何のページか・なぜ記事を書くのか
    ↓
[GUIDE] ルール・コツ・事例（投稿ガイド）
    ↓
[FORM] 記事URL提出フォーム
    ↓
[CONFIRMATION] 提出完了＋次のステップ
```

---

### セクション①：HERO（ファーストビュー）

**目的：** 「ここはフィードバックを得るための場所」と一目で分かり、行動意欲が湧く

| 要素             | 内容                                                                                           |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| **ヘッドライン** | 「あなたの"考えた軌跡"を見せてください」                                                       |
| **サブヘッド**   | BONOで学んだことを実践し、思考の記事を書いて提出すると、プロからのフィードバックを受けられます |
| **CTA**          | 「提出ガイドを読む ↓」（ページ内スクロール）                                                   |

> [!IMPORTANT]
> **Airbnb的ポイント：** 「記事を判定する」ではなく「あなたの成長を見せてほしい」というトーン。ホスト向けリスティング作成ページの「あなたのスペースを共有しましょう」と同じ構造。

---

### セクション②：GUIDE（投稿ガイド）

**目的：** 何を書けば通るのか分かる。不安を解消して記事執筆に着手できる

#### 2-a. ルール（3つの必須項目）

- 投稿ガイドの「ルール」セクションをそのまま表示
- **UIパターン：** アコーディオンではなく**カード型**で3つ並べる（Notice / Before-After / Why）
- 各カードにアイコン + 一言の定義 + 具体例
- 「どれか1つあればOK」を視覚的に強調（バッジ or タグ）

> [!TIP]
> **Airbnb的ポイント：** Airbnbのホスト登録ステップのように、3つの項目を「段階」ではなく「選択肢」として見せる。「全部やらなくていい」安心感。

#### 2-b. NGパターン

- アコーディオンで折りたたむ（デフォルトは閉じた状態）
- ❌/✅ の対比表示で「こう書くとNG」「こう書けばOK」を直感的に見せる

#### 2-c. コツ

- 3つのコツをステップ形式で表示
- 各コツに1行の要約 + 展開すると詳細が見える

#### 2-d. 事例

- OK/NG事例をタブ切り替えまたはカルーセルで表示
- 各事例に「なぜOK/NGか」を1〜2行で添える
- 実際のnote記事へのリンク付き

> [!NOTE]
> **ガイド全体の情報設計：** 上から順に「何を書くか（ルール）→ 何を避けるか（NG）→ うまく書くには（コツ）→ 実例で確認（事例）」。ユーザーの理解を段階的に深める構造。

---

### セクション③：FORM（提出フォーム）

**目的：** 摩擦なく記事URLを提出できる。入力ミスを防ぐ

#### フォーム項目

| #   | 項目名                               | タイプ                           | 必須 | 説明                                 |
| --- | ------------------------------------ | -------------------------------- | ---- | ------------------------------------ |
| 1   | **名前**                             | テキスト                         | ✅   | Slack表示名と同じもの                |
| 2   | **記事URL**                          | URL                              | ✅   | note等の記事URL                      |
| 3   | **この記事で一番伝えたいこと**       | テキストエリア（140字以内）      | ✅   | 後述                                 |
| 4   | **どの項目に該当すると思いますか？** | 複数選択                         | ✅   | Notice / Before-After / Why から選択 |
| 5   | **主に使ったBONOコンテンツ**         | ドロップダウン or フリーテキスト | 任意 | どのコンテンツを実践したか           |

> [!IMPORTANT]
> **項目3「この記事で一番伝えたいこと」が最重要フィールド。**
>
> これはバリデーションを兼ねた設計。ユーザーが「一番伝えたいこと」を140字で言語化する時点で、「やったことの羅列」記事はここで書けなくなる。
>
> Airbnbのリスティング作成で「ゲストに伝えたいこと」を書かせるのと同じ原理。ユーザー自身に「価値の言語化」を促すことで、記事の質を提出前に底上げする。

#### 項目4の意図

- ユーザー自身に「自分の記事がどのルールに該当するか」を自己判定させる
- これにより：
  - ルールの理解が深まる（事前にガイドを読まないと選べない）
  - 判定側のレビュー負荷が下がる（ユーザーの自己申告と実際のギャップで判定しやすい）

#### フォームのUX

- **プレビュー機能：** URLを貼ると、OGP（サムネイル・タイトル）が自動表示される → 「正しいURLか」の確認が視覚的にできる
- **文字数カウンター：** 項目3に残り文字数を表示
- **提出ボタン：** 「記事を提出する」（「送信」ではない。行為に名前をつける）

---

### セクション④：CONFIRMATION（提出完了）

**目的：** 提出した事実を肯定し、次に何が起きるか明確にする

#### 完了画面の構成

```
🎉 記事を提出しました！

あなたの思考の軌跡を共有してくれてありがとうございます。

📋 判定について
- 判定結果はコミュニティSlackでお伝えします
- 通常 ○営業日以内にご連絡します

💡 パスした場合
→ フィードバック権利が付与されます

🔄 改善点がある場合
→ Slackで具体的にお伝えします。修正の参考にしてください

[OK事例をもう一度見る]  [コミュニティに戻る]
```

> [!TIP]
> **Airbnb的ポイント：** 「NG」ではなく「改善点がある場合」という表現。Airbnbがホストに「リスティングを改善しましょう」と導くのと同じ。拒絶ではなく改善の促し。

---

## 3. バリデーション設計（運営側）

### 判定フロー

```
記事URL提出
    ↓
[STEP 1] 自動チェック
  - URLの有効性
  - 記事内にBONO関連キーワードがあるか（簡易スクリーニング）
    ↓
[STEP 2] AI判定（Gemini）
  - 記事本文を読み込み
  - ルール3項目（Notice / B-A / Why）の該当判定
  - NGパターン（羅列 / コンテンツまとめ直し / 感想のみ）の検出
  - ユーザーの自己申告（項目4）との一致度チェック
  - 判定結果 + 改善コメント案を生成
    ↓
[STEP 3] カイさん最終確認
  - AI判定結果をレビュー
  - 必要に応じて修正・補足
  - Slackでユーザーにフィードバック
```

### AI判定プロンプト（将来実装用メモ）

```
以下のルールに基づいて、この記事を判定してください。

## ルール（どれか1つ満たしていればパス）
1. Notice：自分の考え方やデザインがどう変わったかが書かれている
2. Before/After：成果物を気づきを得て自ら修正した過程が伝わる
3. Why：なぜそうしたかを自分の言葉で説明している

## NGパターン（該当したらNG）
- やったことの羅列（何を考えたか不明）
- コンテンツ内容のまとめ直し（自分の思考がない）
- 感想のみ（「勉強になりました」）

## 判定基準
読者がこの記事を読んで「筆者の思考プロセスを追体験できるか？」

## 出力形式
- 判定：✅パス / ❌NG / ⚠️要改善
- 該当項目：Notice / Before-After / Why のどれに該当するか
- 理由：なぜそう判定したか（2〜3文）
- 改善コメント案（NGまたは要改善の場合のみ）
```

## 4. 成功指標（KPI）

| 指標              | 意味                                       | 目標         |
| ----------------- | ------------------------------------------ | ------------ |
| **提出率**        | コンテンツ受講者のうち記事を提出した割合   | 30%以上      |
| **初回パス率**    | 最初の提出でパスした割合                   | 60%以上      |
| **再提出率**      | NGフィードバック後に修正して再提出した割合 | 50%以上      |
| **記事→引用RT率** | パスした記事のうち引用RTされた割合         | カイさん裁量 |

> [!NOTE]
> 初回パス率が低すぎる場合 → ガイドやコツの説明が不十分。ページ側を改善。
> 初回パス率が高すぎる場合 → ルールが甘い、またはユーザーの質が高い。いずれにせよ良い指標。

## 5. 実装優先度

| Phase              | 内容                                                            | 優先度    |
| ------------------ | --------------------------------------------------------------- | --------- |
| **Phase 1（MVP）** | ガイドページ + Google Form等でURL＋伝えたいこと＋該当項目を提出 | 🔴 最優先 |
| **Phase 2**        | OGPプレビュー + 専用フォームページ（Webサイトに組み込み）       | 🟡 次点   |
| **Phase 3**        | AI判定の自動化（Gemini API連携）                                | 🟢 将来   |

> **MVP方針：** Phase 1はNotionページ or Google Formでも即日実装可能。見た目の完成度より「循環を回し始めること」を優先する。
