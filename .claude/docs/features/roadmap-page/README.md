## 概要

Figmaデザインを元にしたロードマップページの実装。
bo-no.designの実際のコースページを反映させる。

## やって欲しいこと

このロードマップのプロトタイプ実装の目的は、ユーザーがまず学ぶべきことをロードマップで認識でき、自分が取り組むコースをチェックすることをやりやすくすることです。
そしてユーザーはロードマップを軸にして物事を進めていきます。

- ゴール：ロードマップページを作り、ロードマップを並べる

ロードマップのトップ画面

- 並べるロードマップ：対象コース、「未経験から転職ロードマップ」
- 「未経験から転職ロードマップ」の作成→これは対象のロードマップ4つが合わさったロードマップです。なので扱いが少し違うものです。
  - これをどのように情報構造として整理して運用すると良いかを決める
  - それをプロトタイプで実装する
- 美しいロードマップ一覧で使うロードマップオブジェクトを作成して並べる
  - 他のレッスンなどと違う概念なのでしっかり違いがわかる見た目
  - 情報構造から相談して決める
    ロードマップの詳細
  - 未経験ロードマップのページを作る
  - パターン１６をベースにデザインはロードマップの詳細ページを作り、各レッスンへのリンクも貼られているようにする
    - パターン１６だが「得られるもの」のセクションに、「このロードマップについて詳しく知る」のボタンリンクは必須。なぜならロードマップの進め方やし指針、なぜやるのか？理解したい層もいるし、理解した方が良いため。
  - レッスンコースへのリンクはきちんとつける。→レッスンの内容と同じなので「レッスンアイコン」画像をしっかり使い、タイトルも揃える

## 実装計画（詳細版）

### クオリティ向上戦略

#### 1. 「一つのことに集中」アプローチ
- 各サブタスクを完全に終えてから次に進む
- 中途半端な状態で次のフェーズに行かない
- 完了条件を明確にして、達成を確認してから進む

#### 2. Agent Teams についての検討

**結論: Agent Teamsは使用しない**

**理由:**
- ユーザーの指摘「一気にやるとパフォーマンスがガタ落ち」
- 並列作業より**順序立てた集中**がこのタスクには適している
- コンポーネント間の依存関係が強い（データ層→コンポーネント→ページ）
- Pattern 16のデザインを一貫して維持する必要がある

**代替アプローチ:**
- 各フェーズを細かく分解し、一つずつ完成させる
- 各タスク後に**ビジュアル確認**を挟む
- 問題を発見したらその場で解決（後回しにしない）

#### 3. 品質チェックポイント
- 各サブフェーズ完了時に動作確認
- TypeScript型エラーがないことを確認
- ブラウザで実際に見た目を確認

## アーキテクチャ決定事項

分析の結果、以下のアプローチを採用：

### ハイブリッドデータアプローチ

```
静的コース設定（TypeScript）
  ├── コースタイトル・説明
  ├── 対象者タグ
  ├── 得られるもの（ベネフィット）
  ├── ステップ構造
  └── lessonSlugs[] ←→ Sanity Lessonと連携

Sanity CMS
  └── Lesson（タイトル、説明、サムネイル、URL）
```

### 理由

- SanityにCourseスキーマが存在しない
- 既存のLessonデータを活用できる
- コース構造は柔軟に変更可能
